## Seção 2 — BUG-2: Range narrativo no briefing (já discutido)

### AS-IS

**Prompt LLM** (`server/routers-fluxo-v3.ts:1923-1938`, Sprint 3 FIX-VIS-U2):
```
REGRA OBRIGATÓRIA SOLARIS (Sprint 3 — FIX-VIS-U2):
- Se houver QUALQUER resposta negativa ("não" / "não sei") no bloco
  <respostas_solaris_onda1>, você DEVE incluir pelo menos 1 gap em
  "principais_gaps" com:
  - "source_type": "solaris"
  - "source_reference": "SOL-NNN" (código da pergunta correlata)
```

**LLM interpretou "pelo menos 1" como "exatamente 1"** e usou range narrativo `"SOL-038 a SOL-049"` para descrever 12 perguntas em 1 entry. Cumpre mínimo legal mas viola contrato (source_reference deveria ser código único).

### Resultado runtime
- 5520001: 1 entry SOLARIS com range "SOL-038 a SOL-049" (12 perguntas comprimidas)
- 5550001: 1 entry SOLARIS com range "SOL-044 a SOL-048" (5 perguntas comprimidas)
- **Cobertura SOLARIS no briefing: ~8% (1/12 ou 1/5 entries de 7-8 totais)**

### TO-BE — Opção MÍNIMA (~5 LOC no prompt)

```diff
// routers-fluxo-v3.ts:1923-1938 (e mesma mudança em :4451 fallback)
REGRA OBRIGATÓRIA SOLARIS (Sprint 3 — FIX-VIS-U2):
- Se houver respostas negativas SOLARIS, você DEVE incluir
- pelo menos 1 gap em "principais_gaps" com source_type="solaris"
+ - **UMA ENTRY POR CATEGORIA SOLARIS AFETADA** (não consolidar em range).
+ - source_reference DEVE ser código único "SOL-NNN" — nunca range
+   "SOL-NNN a SOL-MMM" ou lista "SOL-NNN, SOL-MMM".
+ - Se houver 5 perguntas negativas, gerar 5 entries (ou pelo menos
+   1 entry por sub-categoria — máximo 8 totais segundo Zod schema).
```

### Validação (Zod)

`ai-schemas.ts:193` aceita string livre em `source_reference`. Adicionar validação opcional:
```diff
- source_reference: z.string().optional()
+ source_reference: z.string().regex(/^SOL-\d+$|^[^\s]/).optional()
+   // SOL-NNN exato OU qualquer string sem range/lista
```

**Mas:** validação Zod estrita pode quebrar Zod `.catch()` para outros source_types (rag/cnae/etc). **Recomendação:** mudar só o prompt + adicionar test de regressão que valida formato pós-LLM.

### Severidade & escopo
- **🟡 médio** — degrada UX/rastreabilidade mas não bloqueia produto
- **Classe A cirúrgica** — ~5 LOC no prompt + ~10 LOC teste
- **2 arquivos:** `routers-fluxo-v3.ts:1923, 4451` (2 prompts) + 1 test

## Seção 3 — BUG-3: Gaps orphan `enquadramento_geral`

### AS-IS

**Cadeia:**

```
solaris_questions.risk_category_code = 'enquadramento_geral' (SOL-043, SOL-044, SOL-047)
   ↓ G17-MAX (Sprint 3 FIX-08)
project_gaps_v3 row escrita com source='solaris', risk_category_code='enquadramento_geral'
   ↓ generateRisksAllSources (risks-v4)
risk-engine-v4.ts:547 GATE:
  if (categoria === "enquadramento_geral") {
    console.warn("NO_QUESTION protocol — REGRA-ORQ-29 / Issue #1045");
    continue;  ← NÃO gera risco
  }
   ↓
risks_v4: 0 rows para enquadramento_geral
project_gaps_v3: 3 rows orphan (sem risk_id correspondente)
```

**Comentário inline do engine** (linha 540-547):
> "Caminho típico: iagen-gap-analyzer fallback 'risco_sistemico' → mapTopicToCategory → 'enquadramento_geral' → risco órfão de fundamentação."
> "Solução: gap permanece em project_gaps_v3 (auditoria preservada), mas risco NÃO é gerado em risks_v4."

### Análise

**Issue #1045 era para caminho IAGEN.** Sprint 3 G17-MAX SOLARIS agora também passa por `enquadramento_geral` quando perguntas curadas têm essa categoria — caminho **não previsto** pela Issue #1045.

**Duas interpretações possíveis:**

#### Interpretação (a) — Curadoria errada

SOL-043 ("Estratégia preventiva de contraditório"), SOL-044 ("Conferência da apuração do Fisco"), SOL-047 ("Mapeamento de teses jurídicas") foram **mal-categorizadas** pelo advogado tributarista. Deveriam ter categoria normativa real (ex: `obrigacao_acessoria`, `transicao_iss_ibs`, etc.) em vez de `enquadramento_geral` fallback.

**Fix:** auditoria jurídica re-classificar 3 perguntas em `solaris_questions`:
```sql
UPDATE solaris_questions SET risk_category_code = '<categoria_real>'
WHERE codigo IN ('SOL-043', 'SOL-044', 'SOL-047');
```

**Vantagem:** sem mudança de código. Pure data curation.
**Risco:** decisão jurídica, não técnica — exige advogado.

#### Interpretação (b) — Design intencional (categoria "auditoria sem risco")

Algumas perguntas SOLARIS são preventivas/operacionais — não materializam risco normativo direto, mas merecem registro para evidência de auditoria. Manus sugeriu:
> "campo `risk_suppressed=1` no gap para clareza"

**Fix:**
1. ALTER TABLE `project_gaps_v3` ADD COLUMN `risk_suppressed BOOLEAN DEFAULT 0`
2. Engine setar `risk_suppressed=1` quando categoria é `enquadramento_geral` (em vez de só continuar)
3. UI poderia ter aba "Gaps de auditoria" para mostrá-los separadamente

**Vantagem:** explicita o orphan como intencional.
**Risco:** scope maior (migration + ALTER + UI nova).

### TO-BE — Decisão necessária pelo P.O.

| Opção | Descrição | Escopo |
|---|---|---|
| **A** | Re-curar 3 perguntas SOLARIS para categorias normativas | Data only — 3 UPDATEs SQL |
| **B** | Adicionar `risk_suppressed` flag + UI separada | Classe B — migration + engine + UI |
| **C** | Manter como está (orphan tolerado) | Zero LOC — registrar como design choice em ADR |

**Recomendação técnica:** **(A)** se advogado validar que as 3 perguntas podem ser melhor categorizadas; senão **(C)** + documentar. **(B)** apenas se houver UX prevista para "Gaps de auditoria sem risco".

### Severidade & escopo
- **🟡 médio** — não bloqueia produto, mas cria dados orphan no banco
- **Opção (A):** zero código, exige decisão jurídica
- **Opção (B):** Classe B — ~50 LOC + migration
- **Opção (C):** zero código, exige ADR documentando

## Seção 5 — BUG-5: Q2 IA Gen 0% apesar de respondido

### AS-IS — Mecanismo do cálculo

`server/lib/briefing-confidence-signals.ts:353-354`:
```ts
// Q2 IA Gen — binário (schema atual só grava quando responde).
const q2Respostas = await db.countOnda2Answers(input.projectId);
```

`server/lib/calculate-briefing-confidence.ts:91`:
```ts
// Q2 IA Gen — binário. Se respostas > 0 → 1, senão 0. Total ignorado.
```

**Conta queries `iagen_answers` table:**
```sql
SELECT COUNT(*) FROM iagen_answers WHERE project_id = ?
-- Se > 0 → completude = 100%
-- Se = 0 → completude = 0%
```

### Possíveis causas do bug (4)

#### (1) `countOnda2Answers` query bugada
**Verificar:**
```bash
grep -n "countOnda2Answers" server/db.ts server/lib/briefing-confidence-signals.ts
```

#### (2) Respostas IA Gen salvas em outra tabela
- IA Gen pode ter migrado de `iagen_answers` para tabela nova sem atualizar `countOnda2Answers`
- Histórico: tabela atual era `iagen_answers` desde Z-11

#### (3) Race condition assíncrona
- Briefing calculado ANTES das respostas IA Gen serem persistidas (auto-save assíncrono)
- Próxima geração de briefing pode mostrar % correta

#### (4) Project 5520001 não tem rows reais em `iagen_answers`
- Manus alegou "respondeu 4 perguntas IA Gen" mas talvez não foram persistidas
- Bug pode ser de SAVE, não de COUNT

### TO-BE — Diagnóstico runtime obrigatório antes de fix

Despachar Manus:
```sql
-- Confirma se respostas IA Gen estão persistidas
SELECT COUNT(*) AS total, MIN(createdAt) AS primeira, MAX(updatedAt) AS ultima
FROM iagen_answers
WHERE projectId = 5520001;

-- Esperado se Manus respondeu 4: total = 4
-- Se total = 0: bug é de SAVE (IA Gen não está persistindo)
-- Se total = 4 + timestamps DEPOIS de briefing_generated: race condition
-- Se total = 4 + timestamps ANTES: bug em countOnda2Answers
```

**Sem essa query, qualquer fix é tiro no escuro** (Lição #93).

### Severidade & escopo
- **🔴 crítico** se afeta cálculo de confiança visível ao usuário (UX engana)
- **Triagem:** diagnóstico SQL → 1 das 4 causas → fix específico
