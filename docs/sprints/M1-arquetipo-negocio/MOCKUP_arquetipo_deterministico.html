Notas de design e limites deste mockup

Anexo
  • O mockup cobre o contrato de UX do arquétipo determinístico proposto pelo P.O. — 5 blocos + validação final. Não cobre a descrição livre + LLM de CNAE (restrição #3 preserva fluxo atual de CNAE inferido antes do arquétipo).
  • Cada campo tem declaração visível de EFEITO (caixa verde). Nenhum campo aparece sem efeito — satisfaz o axioma "cada campo deve ter pelo menos uma regra de efeito declarada".
  • Sub-bloco F4-02-sub (regime especial) demonstra progressive disclosure com cauda: regime especial é binário Sim/Não, e o Sim abre um campo fechado enumerado (não texto livre).
  • Bloco 5 Validação tem 2 severidades explícitas: HARD BLOCK (V-01) e WARN (V-02, V-03). Mockup mostra ambos em estados distintos (E = HARD, F = todos OK).
  • Painel lateral expõe o arquétipo em construção como JSON vivo — dá feedback contínuo ao user e cria âncora mental do "resultado sendo construído".
  • Pontos não resolvidos no modelo (ver CRITIQUE-UI-SPEC.md e DECISAO-GO-NO-GO-M1.md): regras de precedência CNAE × respostas do user (B4), fallback para CNAEs não-mapeados (R4), multi-setor com subnatureza array (R5), 7 P0 pendentes para v3.
  • O que o mockup NÃO mostra ainda: (a) fluxo de descrição livre + confirmação de CNAE no início (preservado do atual); (b) campos de identificação (CNPJ, natureza jurídica, nome — ficam antes do Bloco 0 na UX real); (c) casos de natureza multi-setor (saúde+financeiro).
